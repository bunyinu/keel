Security

Trust & security

Keel is repo-local agent state. Your goals live in .keel/ in your repository — not in our training data.

What we store

  • Local (default): state.json, snapshot.md, hook logs — on your machine, in your repo.
  • Keel Cloud (optional): If you run keel cloud link, we store your project state and snapshot to sync across machines. You control the API key.

What we do not do

  • We do not use your goals or snapshots to train models.
  • We do not run your code on our servers (except optional cloud sync API).
  • Hooks run as your user on your machine — same trust model as Claude Code / Cursor hooks.

Hooks & enforcement

Keel hooks can block tool use (dependency installs, edits in read-only mode) and block session end until an acceptance command passes. Review .claude/settings.json, .codex/hooks.json, and .cursor/hooks.json after keel init.

Signed policy (v0.4+)

Keel cryptographically signs the goal policy (title, acceptance criteria, constraints). Default algorithm is ECDSA P-256 (NIST secp256r1).

Important: FIPS-approved algorithm ≠ FIPS 140 validated module. Keel ships pure Rust crypto for portability.

  • Trusted: policy.pub, policy.sig, and policy.md (signed goal only).
  • Secret: policy.key — never commit.
  • Untrusted: snapshot.md progress and failure logs (agent-written).
  • With policy.mode: required, hooks block all tools if the goal is unsigned or tampered.

Keel Cloud access control

  • Dashboard URLs only show a login gate until a valid project API key is entered.
  • API reads/writes require Authorization: Bearer <api_key>.
  • Project creation may require server-side KEEL_CREATE_SECRET.

Data deletion

  • Local: Delete .keel/ and remove Keel hook entries from agent settings.
  • Cloud: Delete the project from your dashboard or contact support for team account deletion.
