Trust & security

Keel is repo-local agent state. Your goals live in .keel/ in your repository — not in our training data.

What we store

  • Local (default): state.json, snapshot.md, hook logs — on your machine, in your repo.
  • Keel Cloud (optional): If you run keel cloud link, we store your project state and snapshot to sync across machines. You control the API key.

What we do not do

  • We do not use your goals or snapshots to train models.
  • We do not run your code on our servers (except optional cloud sync API).
  • Hooks run as your user on your machine — same trust model as Claude Code / Cursor hooks.

Hooks & enforcement

Keel hooks can block tool use (dependency installs, edits in read-only mode) and block session end until an acceptance command passes. That is intentional — review .claude/settings.json, .codex/hooks.json, and .cursor/hooks.json after keel init.

Keel Cloud access control

Cloud projects are keyed, not public-by-link:

  • Dashboard URLs (/dashboard/{id}) only show a login gate. Goal text and snapshots load only after a valid project API key is entered (same key as keel cloud link).
  • API reads and writes (GET/POST/PUT /api/projects/…) require Authorization: Bearer <api_key>. A UUID in the URL is not enough.
  • Project creation may require a server-side create secret (KEEL_CREATE_SECRET) so bots cannot spam new projects.
  • Treat your API key like a password — anyone with it can read and change that project’s cloud state.

Data deletion

  • Local: Delete .keel/ and remove Keel hook entries from agent settings.
  • Cloud: Delete the project from your dashboard or stop syncing; contact support for team account deletion.
